# Spring Boot CRUD Application - Complete Guide

## 📋 Prerequisites

- **Java JDK 17** or higher
- **Maven 3.6+** or use Maven Wrapper (included)
- Any IDE (IntelliJ IDEA, Eclipse, VS Code)
- A web browser

## 📁 Project Structure

```
crud-app/
├── pom.xml
├── users.json (auto-generated)
└── src/
    └── main/
        ├── java/
        │   └── com/example/crud/
        │       ├── CrudApplication.java
        │       ├── controller/
        │       │   └── UserController.java
        │       ├── model/
        │       │   └── User.java
        │       └── service/
        │           └── UserService.java
        └── resources/
            ├── application.properties
            └── static/
                ├── index.html
                ├── style.css
                └── script.js
```

## 🚀 How to Run

### Step 1: Create the Project Structure
Create folders as shown above and copy all the files into their respective locations.

### Step 2: Build the Project
```bash
# Navigate to project directory
cd crud-app

# Build with Maven
mvn clean install

# OR use Maven Wrapper (if available)
./mvnw clean install
```

### Step 3: Run the Application
```bash
# Run with Maven
mvn spring-boot:run

# OR run Maven Wrapper
./mvnw spring-boot:run

# OR run the JAR directly
java -jar target/crud-app-1.0.0.jar
```

### Step 4: Access the Application
Open your browser and go to:
```
http://localhost:8080
```

## 🔧 How CRUD Operations Work

### 1. **CREATE (Add User)**
- **Frontend**: Fill the form and click "Add User"
- **JavaScript**: `POST` request to `/api/users`
- **Java Backend**: `UserController.createUser()` receives the request
- **Service Layer**: `UserService.createUser()` generates ID and saves to `users.json`
- **File**: User added to JSON array in file

**Code Flow:**
```
Form Submit → script.js:createUser() → POST /api/users → 
UserController → UserService → Write to users.json
```

### 2. **READ (View Users)**
- **Frontend**: Page loads automatically
- **JavaScript**: `GET` request to `/api/users`
- **Java Backend**: `UserController.getAllUsers()` handles request
- **Service Layer**: `UserService.getAllUsers()` reads from `users.json`
- **Response**: JSON array of users displayed on page

**Code Flow:**
```
Page Load → script.js:loadUsers() → GET /api/users → 
UserController → UserService → Read users.json → Display
```

### 3. **UPDATE (Edit User)**
- **Frontend**: Click "Edit" button on any user card
- **JavaScript**: 
  - `GET` request to `/api/users/{id}` to load user data
  - Form populates with existing data
  - User modifies and submits
  - `PUT` request to `/api/users/{id}`
- **Java Backend**: `UserController.updateUser()` receives update
- **Service Layer**: `UserService.updateUser()` finds user by ID and replaces data
- **File**: User object updated in JSON array

**Code Flow:**
```
Edit Button → script.js:editUser() → GET /api/users/{id} → 
Populate Form → User Submits → PUT /api/users/{id} → 
UserController → UserService → Update users.json
```

### 4. **DELETE (Remove User)**
- **Frontend**: Click "Delete" button → Confirmation dialog
- **JavaScript**: `DELETE` request to `/api/users/{id}`
- **Java Backend**: `UserController.deleteUser()` receives request
- **Service Layer**: `UserService.deleteUser()` filters out user with matching ID
- **File**: User removed from JSON array and file saved

**Code Flow:**
```
Delete Button → Confirm → script.js:deleteUser() → 
DELETE /api/users/{id} → UserController → UserService → 
Remove from users.json
```

## 📝 Data Storage Explained

### users.json Format
```json
[
  {
    "id": 1,
    "name": "John Doe",
    "email": "john@example.com",
    "age": 30
  },
  {
    "id": 2,
    "name": "Jane Smith",
    "email": "jane@example.com",
    "age": 25
  }
]
```

### How It Works Without Database:
1. **File as Storage**: `users.json` acts like a database table
2. **Jackson Library**: Converts Java objects ↔ JSON automatically
3. **In-Memory Operations**: All reads load entire file into memory
4. **Atomic Writes**: Every change rewrites the entire file
5. **ID Generation**: `AtomicLong` counter ensures unique IDs

### Trade-offs:
✅ **Pros**: Simple, no database setup, portable
❌ **Cons**: Not suitable for large data, no concurrent access control, data lost if file corrupted

## 🔍 API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/api/users` | Create new user |
| GET | `/api/users` | Get all users |
| GET | `/api/users/{id}` | Get single user |
| PUT | `/api/users/{id}` | Update user |
| DELETE | `/api/users/{id}` | Delete user |

## 🛠️ Troubleshooting

**Port 8080 already in use:**
- Change port in `application.properties`: `server.port=8081`

**users.json not found:**
- Application creates it automatically on first run
- Check project root directory

**CORS errors:**
- `@CrossOrigin` annotation already added in controller

**Changes not reflecting:**
- Stop and restart the Spring Boot application
- Clear browser cache or hard refresh (Ctrl+F5)
